# React — Two-Way Binding, Props Drilling & State Lifting

## 1. Two-Way Binding

> **Two-way binding just means adding a `value` prop to the input tag, alongside `onChange`.**

Compare the two patterns:

```jsx
// One-way (uncontrolled-ish): React only listens, doesn't drive the displayed value
<input onChange={(e) => setFormData({ ...formData, name: e.target.value })} />

// Two-way binding: React BOTH reads AND writes the input's value
<input
  value={formData.name}
  onChange={(e) => setFormData({ ...formData, name: e.target.value })}
/>
```

| | One-way (no `value`) | Two-way binding (`value` + `onChange`) |
|---|---|---|
| Who controls what's shown on screen | The DOM/browser itself | React state |
| `onChange` role | Just *notified* of changes | Captures the keystroke AND feeds it back into state, which then re-renders the input with the new `value` |
| Can you reset/clear the input from code? | Not directly | Yes — just set `formData.name` back to `""` in state, and the input updates itself |
| React term | "Uncontrolled" component | **"Controlled" component** |

**Why it matters:** with `value={formData.username}` wired up, the input's displayed text is *always* whatever is currently in state — not whatever the DOM happens to hold. This is exactly what lets `Register.jsx` reset the entire form after submit with a single line:

```js
setFormData({ username: "", email: "", password: "" });
```

Without `value=` bound to each input, this reset would only clear React's internal state — the actual text visible in the input boxes on screen would stay exactly as the user left it. The `value` binding is what forces the DOM to sync back to state.

## 2. Props — One-Way Flow (Parent → Child Only)

> **Props can only ever "drill" from parent to child — like genes passed down a family tree.**

```
App (parent)
  │  passes props down
  ▼
Login / Register (children)
```

- A parent can pass data or functions **down** into its children via props.
- A child has **no direct way** to pass data back up to its parent, or sideways to a sibling component.
- This is called **unidirectional data flow** — data only ever flows in one direction, which is a deliberate design choice in React. It makes apps predictable and easy to debug: if something changed, you know it came from somewhere *above* in the tree, never from below.

### So how does a child ever affect its parent?

By receiving a **function** as a prop (the parent defines the function; the child just *calls* it). Calling a function passed down as a prop **is not** the child sending data "upward" through props — it's the child triggering logic that the parent already owns. This is exactly what `setToggle` and `setUsers` are doing in the code below: `App` owns the actual `toggle`/`users` state and its setters, and simply **hands the setter functions down** as props so children can trigger updates in the parent, without ever "owning" that state themselves.

## 3. State Lifting (a.k.a. "Lifting State Up")

> **State uplifting**: if a piece of data needs to be shared/used by multiple components, move that state **up** to their closest common parent, and pass it down (and pass setters down) as props.

**Why this pattern is needed:**

- `Login` and `Register` both need to be able to **flip between each other** ("Register" link inside `Login`, "Login" link inside `Register`). If each component tried to keep its own local `toggle` state, they'd have no way to coordinate — flipping `Login`'s own internal toggle wouldn't affect `Register` at all, since they're siblings and can't talk to each other directly (per the one-way props rule above).
- **Solution:** the `toggle` state (and the `users` list, since registered users presumably need to be available elsewhere too, e.g. for a login check) is lifted up to `App` — the common parent of both `Login` and `Register`. `App` then passes `setToggle` (and `setUsers`) down as props to whichever component is currently rendered.
- Now, calling `setToggle` from *inside* `Login` or *inside* `Register` updates the **same shared state** living in `App`, and `App` re-renders, deciding which of the two components to show next.

```
        App
   (toggle, users state lives here)
      /              \
setToggle, setUsers   setToggle, setUsers
  passed down            passed down
      /                      \
   Login                  Register
(uses setToggle only)   (uses setToggle AND setUsers)
```

## 4. Code Walkthrough

> **Note:** `App.jsx` itself wasn't included in what you shared — only `Login.jsx`, `Register.jsx`, `main.jsx`, and `index.css`. Based on the props each component expects (`setToggle` in both, `setUsers` in `Register`), `App.jsx` almost certainly looks something like this (shown here so the full picture makes sense):

```jsx
import React, { useState } from "react";
import Login from "./Login";
import Register from "./Register";

const App = () => {
  const [toggle, setToggle] = useState(true);   // true = show Login, false = show Register
  const [users, setUsers] = useState([]);        // lifted here so it can persist across toggles

  return toggle
    ? <Login setToggle={setToggle} />
    : <Register setToggle={setToggle} setUsers={setUsers} />;
};

export default App;
```

This is the concrete example of **state lifting**: `toggle` and `users` live in `App`, and both `Login` and `Register` are handed exactly the setter functions they need — nothing more.

### 4.1 `main.jsx` — Application Entry Point

```jsx
import { createRoot } from "react-dom/client";
import "./index.css";
import App from "./App.jsx";

createRoot(document.getElementById("root")).render(<App />);
```

- `createRoot(document.getElementById("root"))` grabs the real DOM element with `id="root"` (from `index.html`) and converts it into a React root node.
- `.render(<App />)` mounts your entire component tree, starting from `App`, into that root.
- `import "./index.css"` pulls in the global stylesheet so it applies across the whole app.

### 4.2 `Login.jsx`

```jsx
import React from "react";

const Login = ({ setToggle }) => {
  return (
    <div className="page">
      <div className="left">
        <div className="brand-mark">Z</div>
        <div className="left-content">
          <h2>Welcome back</h2>
          <p>Sign in to pick up right where you left off.</p>
        </div>
        <div className="left-footer">
          <span>© 2026 Zoe Corporation.</span>
        </div>
      </div>
      <div className="right">
        <h1>Login</h1>
        <form action="">
          <input type="text" placeholder="Username" autoComplete="username"/>
          <input type="password" placeholder="Password" autoComplete="current-password"/>
          <button>Login</button>
        </form>
        <p>
          Don't have an account?<span onClick={() => setToggle((prev) => !prev)}>Register</span>
        </p>
      </div>
    </div>

```

- **Destructured prop:** `({ setToggle })` — `Login` receives only the `setToggle` function; it doesn't need `users` at all, since it isn't managing registration.
- **Inputs here are *not* controlled** — no `value=` prop, so there's no two-way binding on this form. It's purely visual right now (the `Login` button has no `onClick`/form submission logic wired up yet).
- **The "Register" link:**
  ```jsx
  <span onClick={() => setToggle((prev) => !prev)}>Register</span>
  ```
  - Uses the **functional updater** form (`prev => !prev`) to flip the boolean — the safe pattern regardless of any pending batched updates (see earlier notes on batching).
  - This is the child **calling a function it was given** — not "sending data up" through props. `Login` has no idea *what* `setToggle` actually does internally; it just knows calling it will flip which form is shown, because that's how `App` wired things up.

### 4.3 `Register.jsx`

```jsx
import React, { use, useState } from "react";

const Register = ({ setToggle, setUsers }) => {
  const [formData, setFormData] = useState({username: "", email: "", password: ""});

  const handleChange = (e) => {
    const { name, value } = e.target;
    setFormData((prev) => ({ ...prev, [name]: value }));
  };

  const handleSubmit = (e) => {
    e.preventDefault();
    setUsers((prev) => [...prev, formData]);
    setFormData({ username: "", email: "", password: "" });
    console.log("User registered:", formData);
  }

  return (
    <div className="page">
      <div className="left">
        <div className="brand-mark">Z</div>
        <div className="left-content">
          <h2>Zoe Welcomes You</h2>
          <p>Sign up to get started.</p>
        </div>
        <div className="left-footer">
          <span>© 2026 Zoe Corporation.</span>
        </div>
      </div>
      <div className="right">
        <h1>Sign Up</h1>
        <form onSubmit={handleSubmit}>
          <input
            required
            value={formData.username}
            name="username"
            onChange={handleChange}
            type="text"
            placeholder="Username"
            autoComplete="username"
          />
          <input
            required
            value={formData.email}
            name="email"
            onChange={handleChange}
            type="email"
            placeholder="Email"
            autoComplete="email"
          />
          <input
            required
            value={formData.password}
            name="password"
            onChange={handleChange}
            type="password"
            placeholder="Password"
            autoComplete="new-password"
          />
          <button>Sign Up</button>
        </form>
        <p>
          Already have an account?
          <span onClick={() => setToggle((prev) => !prev)}>Login</span>
        </p>
      </div>
    </div>
  );
};

export default Register;
```

Breaking this down piece by piece:

**a) Local vs. lifted state, side by side:**
- `formData` — **local** state (`useState` inside `Register` itself). It only matters to this component while the user is actively filling the form, so there's no reason to lift it up to `App`.
- `setToggle`, `setUsers` — **lifted** state setters, received as props from `App`, because that data needs to be shared/coordinated with `Login` and (presumably) the rest of the app.

**b) `handleChange` — the reusable field-updater pattern:**
```js
const { name, value } = e.target;
setFormData((prev) => ({ ...prev, [name]: value }));
```
- Same computed-property-name trick from the previous form-handling module (`[name]: value`), so one function updates whichever field fired the event.
- Notice this uses the **functional updater** form (`(prev) => ({ ...prev, ... })`) rather than referencing `formData` directly from the closure — the safer pattern when updates could stack up (see the batching notes).

**c) `value={formData.username}` etc. — controlled inputs (two-way binding):**
- Each input's displayed text is **fully driven by state**, not by the DOM. This is what makes the reset-after-submit actually visible on screen (see Section 1 above).

**d) `required` attribute + CSS validity styling:**
- `required` is native HTML form validation — the browser won't let the form submit if a required field is empty.
- This pairs with CSS rules in `index.css`:
  ```css
  .right form input:not(:placeholder-shown):invalid {
    border-bottom-color: #E05A5A; /* red */
  }
  .right form input:not(:placeholder-shown):valid {
    border-bottom-color: #3EBB9E; /* green */
  }
  ```
  - `:invalid` / `:valid` are pseudo-classes based on the browser's built-in validation (e.g. `type="email"` must contain `@`, `required` fields can't be empty).
  - `:not(:placeholder-shown)` is a deliberate guard — **without it**, an *empty* required field would show as `:invalid` (red) immediately on page load, before the user has typed anything, which looks broken. `:placeholder-shown` is only true when the input is empty *and* showing its placeholder text — so `:not(:placeholder-shown)` means "only apply validity styling once the user has actually started typing."

**e) `handleSubmit` — form submission logic:**
```js
const handleSubmit = (e) => {
  e.preventDefault();
  setUsers((prev) => [...prev, formData]);
  setFormData({ username: "", email: "", password: "" });
  console.log("User registered:", formData);
}
```
1. `e.preventDefault()` — stops the browser's default form behavior, which would otherwise reload the entire page (the classic behavior of an HTML `<form>` submission).
2. `setUsers((prev) => [...prev, formData])` — appends the just-submitted `formData` object onto the **lifted** `users` array living in `App`, using the spread syntax to create a **new array reference** (same underlying rule as objects: mutating an array in place, e.g. `users.push(...)`, would **not** trigger a re-render — you must create a new array).
3. `setFormData({ username: "", email: "", password: "" })` — resets the local form back to empty strings for all three fields, which visibly clears the inputs on screen *specifically because* they're controlled (`value=` bound).
4. `console.log(...)` — a debugging aid to confirm what was captured, since there's no visible confirmation UI yet.

**f) The "Login" link:**
```jsx
<span onClick={() => setToggle((prev) => !prev)}>Login</span>
```
- Mirrors `Login.jsx`'s "Register" link exactly — flips the same shared `toggle` boolean back the other way, switching `App` to render `Login` instead.

## 5. `index.css` — Styling Breakdown

### 5.1 Custom Font & Reset
```css
@font-face {
  font-family: 'font';
  src: url('./font.ttf');
}

* {
  margin: 0;
  padding: 0;
  box-sizing: border-box;
  font-family: 'Helvetica Neue Light';
}
```
- `@font-face` registers a custom local font file (`font.ttf`) under the name `'font'`, which is then specifically applied to `.right h1` (the big "Login"/"Sign Up" heading) for a distinct, branded look — while the rest of the app uses `'Helvetica Neue Light'` as the base font.
- The universal selector reset (`margin: 0; padding: 0; box-sizing: border-box;`) removes default browser spacing inconsistencies and makes width/height calculations predictable (padding and border are included *within* the declared width/height, rather than adding to it).

### 5.2 Two-Panel Layout
```css
.page {
  display: grid;
  grid-template-columns: 1fr 1fr;
  ...
}
```
- `.page` uses **CSS Grid** with two equal-width columns (`1fr 1fr`) to create the classic split-screen auth layout: `.left` (branding/illustration panel) and `.right` (the actual form).
- `.left` uses a diagonal `linear-gradient` (green tones) plus two large, softly-transparent circles (via `::before`/`::after` pseudo-elements) positioned outside the panel's corners for a subtle decorative touch — purely visual, doesn't affect layout since they're `position: absolute`.
- `.right` is a plain white panel, centered content, holding the heading, form, and the "switch to other form" link.

### 5.3 Input Styling — Underline-Only Style
```css
.right form input {
  border: none;
  border-bottom: 2px solid #E4E7EB;
  background-color: transparent;
  transition: border-color 0.15s ease-in-out;
}
```
- Inputs have **no border box at all** — just a bottom underline, a minimalist style consistent with what's been used across your earlier login/registration UI work.
- `:hover` and `:focus` progressively darken/color the underline (`#B8BDC4` on hover, brand green `#3EBB9E` on focus), giving layered visual feedback as the user interacts.
- `:-webkit-autofill` override forces Chrome's yellow autofill background back to white, since Chrome otherwise ignores your `background-color: transparent` once a field is autofilled.

### 5.4 Responsive Breakpoints

| Breakpoint | Behavior |
|---|---|
| `max-width: 900px` (tablet) | `.page` grows to `85%`/`80%` of viewport; heading font sizes shrink slightly |
| `max-width: 650px` (mobile) | **Layout fundamentally changes**: `.page` switches from `grid-template-columns: 1fr 1fr` to a **single column** (`grid-template-columns: 1fr`, `grid-template-rows: auto 1fr`) — the two panels **stack vertically** instead of sitting side-by-side. `.left-footer` is hidden entirely to save space. |
| `max-width: 380px` (small phones) | Further font-size reductions and a smaller `border-radius`, for very small screens |

This is a classic **mobile-first-adjacent responsive pattern**: rather than redesigning from scratch at each breakpoint, the same grid-based structure is reused, and only column arrangement + sizing are adjusted as the viewport shrinks.

## Quick Summary

```
Two-way binding:  value={state} + onChange={...}  →  input is fully controlled by React state

Props:  ALWAYS flow parent → child only (never child → parent, never sibling → sibling)
        A child can still trigger a parent's state update — but only by calling a
        function the parent already owns and passed down. That's not "reverse data flow";
        it's the child invoking logic, not sending data upward through props.

State Lifting:  if 2+ components need to share/coordinate a piece of state,
                move that state up to their closest common parent (here: App),
                then pass the state (or just its setters) down as props.

This Login/Register pair:
  toggle  → lifted to App → decides which form renders
  users   → lifted to App → shared list, appended to on every successful Register submit
  formData→ stays LOCAL to Register, since nothing else needs it
```
