# React ŌĆö `useRef` Hook

## 1. What is `useRef`?

**`useRef`** is a React hook used to directly access and reference **real DOM elements**, or to store any mutable value that needs to persist across renders **without** causing a re-render when it changes.

```js
const inpRef = useRef();
```

According to the official React documentation:

- <cite index="12-1">useRef returns a ref object with a single current property, initially set to the initial value you provide. On the next renders, useRef will return the same object.</cite>
- <cite index="15-1">You can access the current value of that ref through the ref.current property. This value is intentionally mutable, meaning you can both read and write to it ŌĆö it's like a "secret pocket" of your component that React doesn't track.</cite>

**Key differences from `useState`:**

| | `useState` | `useRef` |
|---|---|---|
| Has a setter function? | Yes (`setValue`) | **No** ŌĆö you mutate `.current` directly |
| Changing it triggers a re-render? | Yes | **No** |
| Value persists across renders? | Yes | Yes |
| Typical use | Data that affects what's rendered on screen | DOM element references, or values you need to track but that shouldn't cause visual updates |

> **`inpRef = useRef()` ŌĆö "iska koi setFunction nahi hota hai"**: correct ŌĆö there's no `setRef` equivalent. You update it with plain assignment: `inpRef.current = something`.

---

## 2. The `ref` Attribute ŌĆö Connecting a Ref to a Real DOM Element

```jsx
<input ref={inpRef} type="text" placeholder="Type something..." />
```

When you attach a ref object to a JSX element's `ref` attribute, React automatically does the wiring for you: <cite index="12-1">if you pass the ref object to React as a ref attribute to a JSX node, React will set its current property.</cite>

So after this input mounts:
```js
inpRef.current  // ŌåÆ the actual real DOM <input> element (not a React element, the genuine browser DOM node)
```

### Why this avoids unnecessary re-renders

This is the core reason `useRef` exists as a separate hook from `useState`: <cite index="12-1">changing a ref does not trigger a re-render. This means refs are perfect for storing information that doesn't affect the visual output of your component.</cite> <cite index="15-1">This is what makes it an "escape hatch" from React's one-way data flow.</cite>

If you needed to read an input's current value on every keystroke, a **controlled input** (`useState` + `value=` + `onChange`) would re-render the component on every character typed. With `useRef`, the DOM manages the input's value internally (uncontrolled), and you only **reach in and read** `.current.value` at the moment you actually need it (e.g., on button click or form submit) ŌĆö with **zero re-renders** along the way.

---

## 3. What are `.current` and `.value`?

These are two **separate, unrelated** properties, from two different layers:

| Property | Belongs to | Meaning |
|---|---|---|
| `.current` | The **ref object** returned by `useRef()` | Holds whatever the ref is pointing at ŌĆö here, the real DOM element itself |
| `.value` | The **native DOM `<input>` element** | A standard HTML DOM property ŌĆö the actual text currently typed in that input box |

So `inpRef.current.value` reads as:

```
inpRef            ŌåÆ the ref object       ({ current: ... })
inpRef.current     ŌåÆ the real DOM <input> element that React attached
inpRef.current.value ŌåÆ that DOM element's own native .value property (the typed text)
```

This is plain browser JavaScript, not a React-specific concept ŌĆö `.value` is the same property you'd access with `document.querySelector('input').value` in vanilla JS. `useRef` is simply giving you a stable, React-friendly way to *get* that DOM element in the first place.

---

## 4. Full Basic Example

```jsx
const app = () => {
  const inpRef = useRef();

  const handleClick = () => {
    console.log(inpRef.current.value); // Accessing the value of the input element
  };

  return (
    <div>
      <input ref={inpRef} type="text" placeholder="Type something..." />
      <button onClick={handleClick}>Get Input Value</button>
    </div>
  );
};
```

- Typing in the input causes **no re-renders** at all ŌĆö the DOM handles the keystrokes on its own (this is an **uncontrolled input**, the opposite of the controlled-input pattern from the form-handling notes).
- Clicking the button runs `handleClick`, which reaches into `inpRef.current` (the real `<input>` DOM node) and reads its native `.value` at that exact moment.
- The **only** re-render that happens here is the initial mount ŌĆö clicking "Get Input Value" itself does not cause the component to re-render, since nothing in state changed; only a ref was read.

## 5. `First.jsx` ŌĆö Same Pattern, Single Ref

```jsx
import React, { useRef } from "react";

const First = () => {
  console.log("First component rendered");
  let inputRef = useRef();

  return (
    <div className="center">
      <input ref={inputRef} type="text" placeholder="Enter Anything..." />
      <button className="btn"
        onClick={() => {
          console.log(inputRef.current.value);
        }}
      >
        Click Me
      </button>
    </div>
  );
};

export default First;
```

- Identical pattern to the basic example above: one `useRef()`, one input, one button that logs `.current.value` on click.
- `console.log("First component rendered")` will print **once** when this component first mounts, and will **not** print again on every keystroke ŌĆö proof that the ref-based approach avoids the re-render-per-character behavior a controlled input would have.

## 6. `Second.jsx` ŌĆö Multiple Refs, One Per Field

```jsx
import React, { useRef } from "react";

const Second = () => {
  const productNameRef = useRef();
  const productPriceRef = useRef();
  const productCategoryRef = useRef();

  console.log("Second component rendered...");

  const handleSubmit = (e) => {
    e.preventDefault();
    console.log(productNameRef.current.value);
    console.log(productPriceRef.current.value);
    console.log(productCategoryRef.current.value);
  };

  return (
    <div>
      <form onSubmit={handleSubmit}>
        <input
          type="text"
          placeholder="Enter Product Name"
          ref={productNameRef}
        />
        <input
          type="text"
          placeholder="Enter Product Price"
          ref={productPriceRef}
        />
        <input
          type="text"
          placeholder="Enter Product Category"
          ref={productCategoryRef}
        />
        <button className="btn">Submit</button>
      </form>
    </div>
  );
};

export default Second;
```

### How this differs from the `formData` object-state pattern (earlier module)

- With `useState`, a 3-field form used **one shared object** (`formData`) plus a single `handleChange`.
- Here, with `useRef`, each field gets its **own separate ref variable** ŌĆö `productNameRef`, `productPriceRef`, `productCategoryRef` ŌĆö created with three individual `useRef()` calls. There's no single reusable "ref handler" equivalent to `handleChange`, because refs are attached once at mount and don't fire on every keystroke.
- `handleSubmit`:
  1. `e.preventDefault()` ŌĆö stops the native form submission's page reload.
  2. Reads each field's current DOM value directly at submit time via `.current.value` ŌĆö no state was ever involved, and **no re-render happened while typing**.
- **Trade-off:** this scales linearly and repetitively ŌĆö 3 fields needed 3 separate `useRef()` declarations and 3 separate `ref=` attributes, similar to the "one `useState` per field" problem from the earlier forms module. `Third.jsx` (below) solves this the same way `handleChange` solved it for state.

## 7. The `ref` Attribute Can Itself Be a Callback Function

Before getting to `Third.jsx`, it's worth calling out explicitly: **the `ref` attribute doesn't only accept a ref object from `useRef()` ŌĆö it can also accept a plain function.** This is officially called a **callback ref**, and it's a distinct pattern from `useRef` (though the two are very often used together, as `Third.jsx` shows below).

```jsx
<input ref={(node) => { /* do something with node */ }} />
```

- <cite index="17-1">React will call the ref callback with the DOM element when the component mounts</cite>, passing the actual DOM node as the function's argument.
- In React 17/18, when the element is removed, React calls that same callback again, but with `null`, giving you a chance to clean up.
- **React 19 changed this:** <cite index="22-1">React 19 introduces a cleaner way to handle this ŌĆö you can return a cleanup function directly from a ref callback, just like from `useEffect`</cite>. <cite index="24-1">When a cleanup function is returned, React calls it on unmount instead of calling the ref callback again with `null`.</cite>

```jsx
// React 19 style ŌĆö setup + cleanup in one place, no useEffect needed
<div
  ref={(node) => {
    if (!node) return;
    const chart = new SomeLibrary(node);   // setup, runs on mount
    return () => {
      chart.destroy();                     // cleanup, runs automatically on unmount
    };
  }}
/>
```

**Why this matters / when to reach for it:**

- `useRef()` alone just gives you a **stable box** (`.current`) to store something in ŌĆö it doesn't run any code for you when the element mounts or unmounts.
- A **callback ref** is a *function*, so React actually **calls it for you** at exactly the right moments (mount, and unmount/cleanup) ŌĆö similar in spirit to `useEffect`, but scoped precisely to that one DOM node's lifecycle rather than the whole component's.
- This makes callback refs useful whenever you need to *do something the instant the DOM node exists* ŌĆö measuring it, attaching a third-party library, adding event listeners ŌĆö instead of just holding onto it passively.

`Third.jsx` (below) uses this exact callback-ref mechanism ŌĆö not to run setup/cleanup logic, but simply to manually route each input's DOM node into a shared object, one key at a time.

## 8. `Third.jsx` ŌĆö One Ref Object for All Fields (Callback Refs)

```jsx
import React, { useRef } from "react";

const Third = () => {
  console.log("Third component rendered...");

  let formRef = useRef({});

  const handleSubmit = (e) => {
    e.preventDefault();
    console.log(formRef.current.productName.value);
    console.log(formRef.current.productPrice.value);
    console.log(formRef.current.productCategory.value);
  };

  return (
    <div>
      <form onSubmit={handleSubmit}>
        <input
          ref={(e) => (formRef.current.productName = e)}
          type="text"
          placeholder="Enter Product Name"
        />
        <input
          ref={(e) => (formRef.current.productPrice = e)}
          type="text"
          placeholder="Enter Product Price"
        />
        <input
          ref={(e) => (formRef.current.productCategory = e)}
          type="text"
          placeholder="Enter Product Category"
        />
        <button className="btn">Submit</button>
      </form>
    </div>
  );
};

export default Third;
```

### 8.1 `useRef({})` ŌĆö Ref Holding an Object

```js
let formRef = useRef({});
```

- Instead of one ref per field, `Third` creates a **single ref** whose `.current` is initialized to an **empty object** (`{}`), rather than a single DOM element.
- The plan: manually attach each individual input's DOM node as a **property** on that one shared object ŌĆö `formRef.current.productName`, `formRef.current.productPrice`, `formRef.current.productCategory` ŌĆö so one ref object ends up holding references to *all three* inputs.

### 8.2 Using a Callback Ref to Populate It

```jsx
<input ref={(e) => (formRef.current.productName = e)} ... />
```

This applies exactly the callback-ref mechanism from Section 7, just for a practical purpose: instead of running setup/cleanup logic, each input's callback ref is used purely to **route** that input's DOM node into the shared `formRef.current` object.

- `(e) => (formRef.current.productName = e)` is the function React calls **automatically at mount time**, passing in the actual DOM `<input>` node as the argument (confusingly named `e` here ŌĆö it is **not** an event object, just a parameter name; it would be clearer named `node` or `el`).
- The function body `formRef.current.productName = e` then manually assigns that DOM node onto the shared `formRef.current` object, under the key `productName`.
- The same pattern repeats for `productPrice` and `productCategory`, each writing into a different key of the same shared object.

**End result**, once all three inputs have mounted:
```js
formRef.current = {
  productName: <input DOM node>,
  productPrice: <input DOM node>,
  productCategory: <input DOM node>
}
```

### 8.3 `handleSubmit` ŌĆö Reading Through the Nested Object

```js
console.log(formRef.current.productName.value);
console.log(formRef.current.productPrice.value);
console.log(formRef.current.productCategory.value);
```

- `formRef.current` ŌåÆ the object holding all three DOM node references.
- `formRef.current.productName` ŌåÆ the specific `<input>` DOM node for that field.
- `.value` ŌåÆ that node's native, browser-level current text value.

### 8.4 Why Go Through This Trouble? (`Second` vs `Third`)

| | `Second.jsx` | `Third.jsx` |
|---|---|---|
| Refs created | 3 separate `useRef()` calls, 3 separate variables | **1** `useRef({})` call |
| How inputs connect | `ref={productNameRef}` (object form) | `ref={(e) => (formRef.current.productName = e)}` (**callback/function form**) |
| Scaling to more fields | Every new field = a whole new `useRef()` declaration + variable | Every new field = just one more line inside the object, no new hook call |
| Access pattern | `productNameRef.current.value` | `formRef.current.productName.value` |

`Third.jsx` demonstrates that `useRef` doesn't strictly need to hold a single DOM node ŌĆö it can hold **any mutable value**, including a plain object that you populate manually via callback refs, giving you a single, extensible container for however many field references you need, similar in spirit to how `formData` consolidated multiple `useState` calls into one object earlier.

> Note: the object-of-DOM-nodes approach in `Third.jsx` is a manual pattern assembled by hand here. In real-world large forms, this same general need (many uncontrolled fields, one place to read them all from) is more commonly solved with a dedicated form library, but understanding this manual version is exactly what makes those libraries make sense later.

## 9. `App.jsx` ŌĆö Tab Switching Between the Three Demos

```jsx
import React, { useState } from "react";
import First from "./components/First";
import Second from "./components/Second";
import Third from "./components/Third";

const App = () => {
  console.log("App component rendered...");
  const [active, setActive] = useState("first");

  const handleClick = (e) => {
    console.log(e.target.innerText);
    setActive(e.target.innerText.toLowerCase());
  };

  return (
    <div>
      <div className="nav">
        <button className="nav-btn" onClick={handleClick}>
          First
        </button>
        <button className="nav-btn" onClick={handleClick}>
          Second
        </button>
        <button className="nav-btn" onClick={handleClick}>
          Third
        </button>
      </div>
      <div className="elem">
        {active === "first" && <First />}
        {active === "second" && <Second />}
        {active === "third" && <Third />}
      </div>
    </div>
  );
};

export default App;
```

### Breakdown

- **`active` state** (a `useState` string, defaulting to `"first"`) tracks which of the three demo components should currently be visible ŌĆö this is ordinary state, unrelated to `useRef`, used purely for tab navigation.
- **`handleClick`**:
  ```js
  setActive(e.target.innerText.toLowerCase());
  ```
  - `e.target` is whichever `<button>` was clicked.
  - `.innerText` reads the button's visible text (`"First"`, `"Second"`, or `"Third"`).
  - `.toLowerCase()` converts it to match the state values (`"first"`, `"second"`, `"third"`).
  - This is a neat shortcut: instead of writing three separate `onClick` handlers (one per button, each hardcoding its own string), all three buttons share **one** `handleClick`, and the button's own text tells the handler which tab to activate ŌĆö conceptually similar to the `[e.target.name]` computed-key trick from the earlier forms module, just applied to text content instead of a `name` attribute.
- **Conditional rendering:**
  ```jsx
  {active === "first" && <First />}
  {active === "second" && <Second />}
  {active === "third" && <Third />}
  ```
  - Only the component whose condition is `true` gets rendered; the other two evaluate to `false` and render nothing (`false` is simply not rendered by React).
  - **Important consequence for `useRef`:** because the non-active components aren't rendered at all (they're fully unmounted, not just hidden with CSS), each time you switch tabs and switch back, that component **remounts from scratch** ŌĆö its `useRef()` calls run again and start with a fresh `.current`, and any previously typed input values are lost. This is different from just hiding a component with `display: none`, which would keep it mounted and preserve its state/refs.
  - This also explains why each child's console log (`"First component rendered"`, `"Second component rendered..."`, `"Third component rendered..."`) only fires when **that specific tab** is switched to ŌĆö not on every click of `App`'s own nav buttons in general.

## Quick Summary

```
useRef(initialValue) ŌåÆ { current: initialValue }   (one stable object across renders)

Changing ref.current            ŌåÆ NO re-render
Changing state via useState     ŌåÆ DOES trigger a re-render

<input ref={myRef} />                     ŌåÆ object-ref form: React sets myRef.current
                                              to the DOM node automatically at mount
<input ref={(node) => { ... }} />         ŌåÆ callback-ref form: React CALLS your function
                                              with the DOM node at mount (and with null at unmount)

.current  ŌåÆ belongs to the REF object (what it's pointing at)
.value    ŌåÆ belongs to the native DOM <input> element (the typed text)

Use useRef when:
  - You need to read a DOM element imperatively (focus, measure, read value on demand)
  - You want to avoid re-rendering on every keystroke (uncontrolled inputs)
  - You need to persist a mutable value across renders that should NOT affect the UI

Conditional rendering ({cond && <Comp />}) unmounts non-matching branches entirely ŌĆö
refs (and all state) inside an unmounted component reset the next time it remounts.
```
